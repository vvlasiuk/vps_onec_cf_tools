# onec_ctx — підготовка контексту з конфігурації 1С

Офлайн-інструмент: з конфігурації 1С (вивантаженої у файли) готує компактне
подання для читання LLM через MCP / API `vps_api`. **Окремий проект**, не
частина `vps_api`.

## Що вже працює (перевірено на реальній УТП, платформа 8.3.9)
- **Розбір Form.bin** (звичайні форми) власним кодом на `onec_dtools`.
  v8unpack 1.2.6 тут непридатний — падає на нестиснутих записах контейнера.
- **Розбір BSL**: сканер, що коректно знаходить межі процедур (не плутається
  на `КонецПроцедуры` в рядках/коментарях, на `#Если`, директивах).
- **Кістяк + винесення тіл**: директиви, сигнатури, `Перем`, регіони,
  доккоментарі лишаються; тіла довші за поріг виносяться в сусідні `.bodies/`.
- **Індекс визначень** (`symbols.json`): символ -> модуль + `Экспорт`.
- **Інспектор .cf** — окремий режим для «чужих» конфігурацій без бази.

## Вхід (основний шлях)
Каталог «Выгрузить конфигурацию в файлы» з Конфігуратора. Дає `.bsl` для
об'єктів/менеджерів/спільних/керованих форм; звичайні форми — у `Form.bin`
(інструмент розпаковує їх сам).

## Запуск
```
pip install -r requirements.txt
cd src
# основний конвеєр -> один SQLite-маніфест:
python -m onec_ctx.cli process <каталог_вивантаження> <шлях>\manifest.sqlite
# діагностика контейнера .cf:
python -m onec_ctx.cli inspect <файл.cf>
```

## Структура коду
```
src/onec_ctx/
    cf/unpack.py     — контейнер .cf/Form.bin: розпакування, інспекція
    cf/form.py       — витяг тексту модуля зі звичайної форми (Form.bin)
    bsl/scanner.py   — лексер BSL: класифікація рядків, межі процедур
    bsl/skeleton.py  — кістяк модуля + винесення тіл
    inventory.py     — обхід дерева вивантаження, збір модулів та індексу
    cli.py           — точка входу (inspect, process)
output/              — артефакт (у .gitignore)
```

## Принципи
- Реконструкція назад НЕ підтримується (артефакт односпрямований, для читання).
- Джерело істини — жива конфігурація; артефакт — похідне.
- Розбір BSL — справжнім сканером, не регекспами по всьому тексту.
- Колізії імен у модулі (дубль під `#Если`) — виявляються й фіксуються в лог.
- Зміни в коді застосовує лише користувач; інструмент готує дані.

## Відомі доробки (наступні кроки)
- Кістяк звичайної форми зараз має ім'я `Form.bin` — перейменувати на `Form.bsl`.
- Індекс: нарізка за цінністю (експорт спільних модулів окремо).
- Поріг god-модуля з автоподілом на файл-на-процедуру.
- Витяг дерева елементів звичайної форми (запис `form`) — за потреби.

## Артефакт: єдиний SQLite-маніфест
Замість дерева з десятків тисяч файлів увесь похідний матеріал — в одному
`manifest.sqlite`. `vps_api` відкриває його read-only й віддає зрізи одним
індексованим SELECT (жодного парсингу в рантаймі, жодного обходу диску).

Схема:
- `meta`    — коли/з чого/чим згенеровано (свіжість артефакту).
- `modules` — по модулю: роль, лічильники, `skeleton_full`, `skeleton_compact`.
- `symbols` — по процедурі: name, kind, is_export, module_path, sig, рядки,
  `body` (текст цілої процедури). Індекси по name та (module_path, is_export).
- `collisions` — виявлені дублі імен у модулі.

Три рівні кістяка (щоб god-модуль не роздував контекст):
- «зміст»   — `SELECT name,kind,is_export FROM symbols WHERE module_path=?`
- «компакт» — `modules.skeleton_compact` (сигнатури без доккоментарів)
- «повний»  — `modules.skeleton_full`

Типові запити vps_api (кожен — один SELECT):
- де оголошено X:  `... FROM symbols WHERE name=? [AND is_export=1]`
- тіло процедури:  `SELECT body FROM symbols WHERE name=? AND module_path=?`

Оновлення свіжості: генератор перегенеровує файл цілком (атомарна підміна);
згодом сигнал «артефакт оновлено» можна слати через RabbitMQ, щоб vps_api
переоткрив БД без рестарту (наступна ітерація).
